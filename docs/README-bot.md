# awg-bot — Telegram-бот управления AmneziaWG (хост бота)

Telegram-бот управления AmneziaWG: роли админ/клиент/друг, инвайты, периоды
подписки с блокировкой трафика при истечении, приостановка подписки («в отпуск»)
с аварийным email-выходом, накопительная статистика, двусторонняя синхронизация
с приложением Amnezia. Мониторинг железа бот снимает сам — локально, с того же
хоста (co-located архитектура).

Документ самодостаточен: всё для установки и эксплуатации бота — ниже.

---

## 1. Роль компонента

Бот ходит в докерный контейнер AmneziaWG (`docker exec`), правит `awg0.conf` /
`clientsTable`, управляет блокировками через iptables внутри контейнера, ведёт
состояние в собственной SQLite-БД и общается с пользователями по Telegram Bot API.

Бот **co-located** с awg-контейнером — стоит на том же хосте и управляет им
локально; по сети до контейнера не ходит. Единственный внешний канал —
Telegram Bot API (плюс, если включён email-выход, исходящий IMAP/SMTP к вашему
почтовому серверу). Топологию установщик определяет сам из живого контейнера.

---

## 2. Требования

- VPS с уже развёрнутой докерной AmneziaWG (контейнер `amnezia-awg2`).
- Python **3.12+** (используется `enum.StrEnum`; сборка ведётся и тестируется на
  3.12 — не полагайтесь на голый минимум 3.11).
- `root` (нужен `docker exec` + управление iptables внутри контейнера).
- Исходящий доступ к Telegram Bot API.

---

## 3. Быстрый старт

Поставка — два файла рядом: `awg-bot.tgz` (код) и `awg-bot-install.sh`
(установщик). Архив вручную распаковывать не нужно — установщик сделает всё сам:

```bash
sudo bash awg-bot-install.sh         # интерактивная установка «в одно окно»
```

Установщик развернёт код в `/opt/awg-bot`, проведёт по шагам (топология,
секреты), соберёт валидный конфиг, поставит и запустит сервис, заведёт команду
`awg-bot` и удалит за собой архив с установщиком. Дальнейшее управление —
командой `awg-bot` (`status`, `logs`, `reconfigure`, `update`, `backup`,
`restore`, `uninstall`). Логи: `awg-bot logs`.

## 4. Что делает установщик (`awg-bot-install.sh` → `awg-bot`)

| Шаг | Действие |
|-----|----------|
| Preflight | Проверяет `root`, ищет Python 3.12+ (при отсутствии предлагает поставить через `apt`/deadsnakes PPA). |
| Код | Bootstrap распаковывает `awg-bot.tgz` в `/opt/awg-bot`, симлинкует команду `awg-bot`, передаёт управление `awg-bot reconfigure --first-run`. |
| venv | Создаёт `venv`, ставит `requirements.txt`. |
| Каталоги FHS | `/etc/awg-bot/conf` (конфиг), `/etc/awg-bot/env` (секреты, `600`), `/var/lib/awg-bot` (БД+бэкапы, `700`). |
| Конфиг | Копирует шаблоны `conf/*.yaml` (только отсутствующие) и **пишет валидные значения** из ответов (топология). |
| Секреты | Спрашивает `BOT_TOKEN`, `ADMIN_ID` → `/etc/awg-bot/env`; опционально запускает `manage_secrets.py`. |
| Email-выход | Опционально настраивает аварийный email-выход из приостановки (§6a) — ящик, пароль, алиас, интервал. |
| Валидация | `config.validate()` — при нехватке параметра падает **до** запуска, с указанием, что и где задать. |
| Сервис | Генерирует и ставит `systemd`-юнит `awg-bot`, `enable --now`. |

Установщик — greenfield-only. Поверх **существующей** установки он ничего не
переписывает молча: обнаружив рабочий venv в `/opt/awg-bot`, предлагает меню —
обновить код из этой поставки (`update`), восстановить из резервной копии
(`restore`), удалить бота (`uninstall`) или выйти. Перенастройка живой
установки — `sudo awg-bot reconfigure`: тот же визард, текущие значения
предлагаются дефолтами (Enter — оставить как есть).

## 5. Модель конфигурации

Два раздельных, предсказуемых источника:

- **Топология и правила → yaml** в `/etc/awg-bot/conf`. Это источник истины.
  Установщик генерит валидные значения под ваши ответы; в эксплуатации правите
  файлы руками **на холодную**. Живут в `/etc`, поэтому **переживают обновление
  кода** (код — в `/opt/awg-bot`, это разные каталоги).
- **Секреты → env** в `/etc/awg-bot/env` (`600`). Только чувствительное
  (`BOT_TOKEN`, `ADMIN_ID`, ключи шифрования бэкапов). Для топологии env **не
  используется**.

Приоритет для деплой-значений: **активная строка yaml → хардкод-дефолт**.
Env-«люка» для топологии нет намеренно — вместо него `config.validate()` при
старте падает с сообщением вида:

```
Не заданы обязательные параметры конфига: app.yaml: network.server_host;
```

Где читается конфиг: `config.py` резолвит `/etc/awg-bot/conf`, если каталог
существует (иначе — `./conf` для запуска из исходников). Переопределить —
`AWG_BOT_CONF_DIR`. Аналогично `AWG_BOT_DATA_DIR` для данных, `AWG_BOT_ENV` для env.

## 6. Справочник конфигурации

Все файлы самодокументированы комментариями. Ниже — назначение и ключевые
параметры каждого.

### `app.yaml` — инфраструктура

| Ключ | Назначение |
|------|-----------|
| `timezone` | Часовой пояс бота (по умолч. `Europe/Moscow`, UTC+3 без летнего времени). |
| `docker.container` / `.interface` / `.awg_dir` | Имя контейнера, WG-интерфейс, каталог Amnezia внутри. |
| `network.subnet_prefix` / `.subnet_cidr` / `.ip_host_start` / `.ip_host_end` | Подсеть VPN и диапазон выдаваемых адресов (сервер занимает `.0`). |
| **`network.server_host`** | **Деплой-ключ, обязателен.** Внешний адрес awg-сервера. |
| **`network.server_port`** | **Деплой-ключ, обязателен.** Порт awg-сервера. |
| `client_config.*` | Параметры клиентского `vpn://`: DNS, MTU, keepalive, allowed_ips, имя сервера. |
| `online_handshake_seconds` | Порог «онлайн» по свежести handshake (выравнен с интервалом опроса). |
| `scheduler.*` | Интервалы фоновых задач (опрос трафика, проверка сроков, мониторинг, месячный сброс, автобэкап, дебаунс вотчдога). |
| `misfire_grace.*` | Льготные окна APScheduler: насколько опоздавший запуск ещё выполняется. |
| `missing_sweeps_threshold` | Сверок подряд без peer'а, прежде чем считать app-устройство удалённым. |
| `history.*` | Срок хранения аудит-истории и параметры фоновой очистки. |

### Блок `monitoring` (app.yaml) — локальный мониторинг хоста

Бот co-located с awg-контейнером, поэтому метрики железа (CPU/RAM/диск) читает
сам — из /proc и statvfs, каждый тик монитора. Агентов, стуков и сетевых
приёмников нет.

| Ключ | Что делает |
|---|---|
| `alert_streak` | Замеров подряд для срабатывания и отбоя ресурс-алерта (гистерезис). |
| `service_failure_alert_loud` | Громкий алерт при устойчивом простое awg-сервиса. |
| `service_failure_alert_minutes` | Порог устойчивого простоя, минут. |

### Бизнес-правила

| Файл | О чём |
|------|-------|
| `subscription.yaml` | Периоды подписки, метки, пороги уведомлений об истечении. |
| `limits.yaml` | Лимиты трафика в календарном месяце, разовая доп.квота, порог пред-уведомления. |
| `grace.yaml` | «Продли на пару недель»: самостоятельная отсрочка (только годовой период). |
| `pause.yaml` | Приостановка подписки клиентом («в отпуск», только годовой период). Единственный лимит — суммарный за период (`pause_max_total_days`, по умолч. 28); длительность одного входа клиент выбирает сам. |
| `email.yaml` | Аварийный email-выход из приостановки (§6a): IMAP/SMTP-серверы, адрес-алиас, интервал опроса, длина кода. Заполняет визард; пустые поля = фича выключена. |
| `quiet_hours.yaml` | Тихие часы: ночью уведомления без звука (окно может пересекать полночь). |
| `resource_alerts.yaml` | Алерты владельцу о загрузке CPU/RAM/диска хоста (локальные замеры, с гистерезисом). |
| `bot_identity.yaml` | Публичные имя и описания бота (ставятся при каждом старте через Bot API). |

## 6a. Аварийный email-выход из приостановки

Опциональная страховка от «запирания в паузе»: если клиент заходит в Telegram
только через этот же VPN, то, поставив подписку на паузу, он теряет доступ и к
боту — и снять паузу сам не может. Email-выход даёт независимый канал: при входе
в паузу клиент получает одноразовый код (8 символов, алфавит без похожих 0/O и
1/l) и, заперевшись, отправляет письмо с этим кодом **в теме** на настроенный
адрес. Бот исходяще опрашивает ящик по IMAP, находит код, снимает паузу,
уведомляет клиента в Telegram и отвечает письмом «Код принят».

Свойства (важно понимать при настройке):

- **Портов на хосте не открывается.** Бот только сам исходит на почтовый сервер
  (IMAP — опрос, SMTP — ответ об успехе), как браузер открывает сайт.
- **`From` письма — не аутентификация** (спуфится): единственный секрет — код,
  известный лишь тому, кто ставил паузу. Тело письма не читается вообще.
- **Невалидный код игнорируется молча**: письмо помечается прочитанным, ответа
  нет — по реакции нельзя понять, существует ли код.
- **Код одноразовый** и живёт только пока активна пауза; снять по почте можно
  только паузу, которую клиент поставил себе сам, — не административную.
- **Интервал опроса — не менее 60 сек**: меньшее значение бот сам откатывает к 60.

**Включение**: `sudo awg-bot reconfigure` → шаг «Настроить аварийный email-выход
из приостановки». Визард спрашивает адрес ящика (для `@icloud.com` серверы
IMAP/SMTP подставляются сами, иначе — ручной ввод), пароль под маской (для iCloud —
app-specific password с `account.apple.com`), опциональный адрес-алиас для писем
от клиентов (Enter — сам адрес ящика) и интервал опроса. Логин/пароль уходят в
`/etc/awg-bot/env` (600), остальное — в `conf/email.yaml`. Повторный заход в
визард предлагает отключить email-выход (стерев настройки) или поменять параметры.

Фича активна, только когда заданы **и** креды в env, **и** хосты в `email.yaml`;
иначе поллер просто не стартует, бот работает как обычно.

Помимо email-выхода у паузы есть ещё два выхода: кнопка «Возобновить» у клиента
и «▶️ Вывести из приостановки» в карточке клиента у администратора.

## 6b. SSH-доступ к хосту из туннеля (только устройства админа)

Бот открывает SSH к хосту через VPN **только устройствам администратора** —
всем остальным пирам SSH-к-хосту закрыт. Фильтр трёхслойный:

1. **Хостовый гейт** — `install/harden_firewall.sh` вносит в SSH-вайтлист хоста
   **bridge-подсеть контейнера** (например `172.29.172.0/24`): именно её хост
   видит как источник туннельного SSH, потому что контейнер маскарадит пиров в
   свой bridge-адрес. VPN-подсеть (`10.8.1.0/24`) до хоста не доезжает —
   правило с ней в хостовом вайтлисте было бы мёртвым.
2. **Пер-пирный фильтр** — цепочка `AWGBOT_SSH` в FORWARD контейнера (там
   исходный `10.8.1.x` ещё настоящий): ACCEPT SSH-к-хосту с адресов устройств
   админа, DROP остальным. Накладывается при создании админского устройства и
   реассертится ботом (старт, монитор-цикл, рестарт контейнера).
3. **Fail-closed** — отдельная `PostUp`-строка в `awg0.conf` ставит глухой DROP
   на SSH-к-хосту в момент подъёма `awg0`, до бота. Окно «контейнер взлетел,
   бот ещё не реассертил» никого не пускает; ACCEPT'ы админам добавляет бот.

**Куда ходить:** из туннеля — на **amn0-шлюз** (адрес хоста на bridge-сети
контейнера, обычно `172.29.172.1`). На **внешний IP хоста из-под туннеля зайти
нельзя** — это физика WireGuard: адрес эндпоинта всегда маршрутизируется мимо
туннеля (иначе не работал бы сам хендшейк), поэтому такой трафик уходит через
физический интерфейс клиента и до фильтров не доходит. Вне туннеля — обычный
вход на внешний IP по правилам хостового фаервола.

**Порт** задаётся в `conf/app.yaml` → `network.ssh_port` (по умолчанию 22) и
должен совпадать с портом в `/etc/ssh/sshd_config`; его читают и
`harden_firewall.sh`, и пер-пирный фильтр.

Два замечания по хосту (вне зоны ответственности бота, но typические грабли):

- **Один владелец правил на SSH-порт.** Если на хосте параллельно живут `ufw` и
  nftables-таблица harden'а — пакет должен пройти оба; вайтлист, внесённый в
  один слой, не спасает от DENY в другом. Ведите правила в одном месте (и
  добавьте bridge-подсеть туда же).
- **`Match Address` в sshd_config.** Если root-вход ограничен по адресам,
  туннельный источник — это bridge-подсеть контейнера (не `10.8.1.0/24`); чтобы
  пускать root из туннеля, добавьте её в разрешающий `Match Address`-блок.

## 7. Секреты (`/etc/awg-bot/env`)

Файл `600`, только чувствительные значения. Обязательные:

| Ключ | Значение |
|------|----------|
| `BOT_TOKEN` | Токен бота от @BotFather. |
| `ADMIN_ID` | Telegram ID администратора (число; узнать у @userinfobot). |

Опциональные (шифрование бэкапов, ровно один из двух):

| Ключ | Значение |
|------|----------|
| `BACKUP_KEY` | base64 32-байтного случайного ключа (храните отдельно — без него бэкап не восстановить). |
| `BACKUP_PASSPHRASE` | Секретная фраза (ключ выводится argon2id; соль — в самом файле бэкапа). |

Опциональные (аварийный email-выход, §6a; **не заполнять вручную** — пишет визард
`awg-bot reconfigure`):

| Ключ | Значение |
|------|----------|
| `EMAIL_RESUME_LOGIN` | Логин почтового ящика (адрес). |
| `EMAIL_RESUME_PASSWORD` | Пароль ящика (для iCloud — app-specific password). |

Оба пусты → email-выход выключен.

Заполнять удобнее интерактивно — `manage_secrets.py` (выбор «случайный/из
пассфразы» для ключа шифрования):

```bash
cd /opt/awg-bot && ./venv/bin/python -m tools.manage_secrets
```

## 8. systemd и пути

Установщик генерирует `/etc/systemd/system/awg-bot.service`:

```ini
[Service]
User=root
WorkingDirectory=/opt/awg-bot
EnvironmentFile=/etc/awg-bot/env
Environment=AWG_BOT_CONF_DIR=/etc/awg-bot/conf
Environment=AWG_BOT_DATA_DIR=/var/lib/awg-bot
ExecStart=/opt/awg-bot/venv/bin/python -m awgbot
Restart=on-failure
```

Точка входа — пакет `awgbot` (`WorkingDirectory` = каталог с `awgbot/`). Форграунд
для отладки — `./run.sh` (тот же `python -m awgbot`, без systemd).

Раскладка на диске:

| Путь | Что |
|------|-----|
| `/opt/awg-bot` | Код + `venv`. Перезаписывается при `update`. |
| `/etc/awg-bot/conf` | yaml-конфиг. Правится руками, `update` не трогает. |
| `/etc/awg-bot/env` | Секреты (`600`). `update` не трогает. |
| `/var/lib/awg-bot` | `bot.db` + `backups/`. **Не терять** — там приватные ключи bot-устройств. |

Управление: `systemctl {status,restart,stop} awg-bot`, логи `journalctl -u awg-bot -f`.

## 9. Обновление и удаление

```bash
sudo awg-bot update                     # архив awg-bot-update.tgz рядом
sudo awg-bot update ./awg-bot-new.tgz   # или явный путь
```

`update` останавливает сервис, заменяет код и зависимости из архива,
переустанавливает юнит, **не трогая** `conf`/`env`/данные (если явно не
согласиться на их удаление), валидирует конфиг и перезапускает. Работает
благодаря разнесению код↔конфиг по разным каталогам.

```bash
sudo awg-bot uninstall                  # снять сервис
```

Снимает и отключает сервис, удаляет юнит. Каталог кода, `/etc/awg-bot` (секреты)
и `/var/lib/awg-bot` (БД!) удаляются **только по явному подтверждению** — по
умолчанию сохраняются.

### 9a. Автообновление из GitHub-релизов

Бот сам следит за новыми версиями в **публичном** релиз-репо
(`justSunny12/awg-bot`) и умеет обновляться по кнопке. Токены/секреты не нужны —
релизы и ассеты качаются анонимно; sha256 берётся из `assets[].digest` (GitHub
считает его сам).

Как работает: раз в сутки (по умолчанию 10:00 в TZ бота) и при каждом старте бот
смотрит список релизов, находит **следующую ступень** за установленной версией
(`awgbot/__version__.py`) и, если она есть, шлёт админу уведомление с changelog
этой версии (тело релиза в сворачиваемой цитате) и кнопками **Обновить /
Скрыть / Не уведомлять**. Идём строго по одной версии вверх — каждый changelog
показывается ровно один раз (дедуп по тегу в БД), промежуточные не
проскакивают. «Обновить» скачивает ассет `awg-bot.tgz` следующей версии, сверяет
sha256 и запускает `awg-bot update` отдельным транзиентным юнитом
(`systemd-run`), чтобы апдейтер пережил остановку сервиса; после рестарта
проверка повторяется и предложит следующую ступень, если она есть. «Скрыть»
убирает сообщение; «Не уведомлять» глушит автопроверку (и суточную, и
стартовую) — при этом ручная кнопка **⬆️ Обновление бота** в админ-меню остаётся
живой и всегда показывает следующую доступную версию.

Если установленной версии нет среди тегов релизов (локальная/кастомная сборка) —
обновления **полностью выключены**: ни уведомлений, ни доступного апдейта.

Настройки (необязательные) — `conf/updates.yaml`: `repo`, `asset_name`,
`poll_hour`/`poll_minute`. По умолчанию всё преднастроено.

### 9b. Процесс релиза (для мейнтейнера)

1. Бампнуть версию в `awgbot/__version__.py` (например `1.2.0`) — она **должна**
   совпасть с тегом релиза без `v`.
2. Собрать поставку: `./build_release.sh` → `dist/awg-bot.tgz`.
3. Создать GitHub-релиз с тегом `vX.Y.Z`; **тело релиза = changelog этой версии
   без общего заголовка** (тег несёт шапка уведомления). Приложить ассет
   `awg-bot.tgz` — sha256 GitHub посчитает сам.
4. Публикация публичного релиза = бот у всех предложит обновиться в течение суток
   (или сразу после их следующего рестарта).

## 10. Эксплуатация

- **Крипто-материал сервера** (обфускация, ключ сервера, psk, порт) читается
  **live из контейнера** и кэшируется (TTL 10 мин + инвалидация вотчдогом);
  генерация клиентского конфига обычно не трогает docker вообще.
- **Блокировки — в БД.** iptables-DROP'ы эфемерны; после рестарта контейнера бот
  переналагает их автоматически (детект по `StartedAt`).
- **Мутации конфига сериализованы** (`mutation_lock`): параллельные добавления
  устройств не теряют peer'ов и не выдают одинаковые IP.
- **Планировщик навёрстывает пропуски**: месячный сброс и автобэкап догоняются на
  старте, если бот лежал через границу месяца.
- **Вотчдог гибридный**: inotify + страховочная mtime-сетка (~10 сек худшая
  задержка подхвата внешних изменений в приложении Amnezia).
- **Поллер email-выхода** (§6a) стартует только при полностью заданной
  конфигурации (креды + хосты); трафик исходящий, интервал — из
  `poll_interval_sec` (не менее 60 сек).
- **Бэкапы** пишутся автобэкапом (1-го числа) и по кнопке; при заданном секрете
  шифруются (SecretBox), открытых копий на диск не ложится. Единственная копия
  конфигов Amnezia вне контейнера — это бэкап. Расшифровка — `python -m tools.restore_backup`
  на awg-хосте.
- **Время везде UTC+3.** Сервер живёт в UTC, бот конвертирует.

## 11. Траблшутинг

| Симптом | Причина / что делать |
|---------|----------------------|
| При старте: `Не заданы обязательные параметры конфига: …` | Впишите указанные ключи в `/etc/awg-bot/conf` (это и есть адрес ошибки) и перезапустите. |
| При старте: `Не заданы обязательные секреты: …` | Заполните `/etc/awg-bot/env` (или запустите `manage_secrets.py`). |
| Сервис не активен | `journalctl -u awg-bot -e` — смотрите трейс. Частое: недоступен docker/контейнер, неверный `BOT_TOKEN`. |
| В логах нет «Вотчдог подключён…» | inotify не пробил namespace контейнера; страховочная mtime-сетка (~10 сек) всё равно подхватит изменения. |
| Клиент не «онлайн» сразу | «Онлайн» определяется по свежести handshake (`online_handshake_seconds`), значение может отставать до интервала опроса. |
| После `docker restart` контейнера пропали блокировки | Нормально: бот переналожит их на следующем цикле мониторинга (`monitor_minutes`). |
| Письмо с кодом выхода из паузы «не срабатывает» | Код должен быть **в теме** письма (тело не читается); проверьте, что письмо ушло на адрес из предупреждения при постановке на паузу, и подождите интервал опроса (`poll_interval_sec`, мин. 60 сек). В логах ищите `email_resume`. Невалидный/уже использованный код игнорируется молча — это штатно (§6a). |

## 11a. Восстановление доступа и app-устройства

Здесь три РАЗНЫХ сценария — не путайте их.

### A. Восстановить управление сервером из приложения (по ссылке полного доступа)

Полезно, если приложение переустановили/сменили устройство, а сервер хотите
снова администрировать из Amnezia (пересоздать, сменить протокол и т.п.).
«Полный доступ» экспортируется ссылкой `vpn://…`, которая несёт управляющие
креды сервера.

**Как получить ссылку полного доступа** (на устройстве, где доступ ещё есть):
на **главном экране** → **«Поделиться»** (внизу) → **«⋮»** (три точки, правый
верх) → **«Поделиться»** → выбрать сервер → **«Скопировать»** (или сохранить
`amnezia_config.vpn`). Затем на новом устройстве добавить сервер по этой ссылке.

**Критично про сеть — и это ПЕРЕСМОТР нашей прежней рекомендации.** Раньше мы
советовали пускать SSH сервера только с адреса VPN-туннеля. **Так делать не надо** —
это загоняет в дедлок (ниже). Правильно: вайтлист **ваших постоянных IP** (дом,
офис, выход вашего VPN); нет статики — DynamicDNS/аналог; в идеале эти IP — за
пределами РФ и маршрутизируются через ваши туннели туда (чтобы природа
управляющего трафика не выдавала себя). SSH держите на нестандартном порту.

Почему это критично именно здесь: развёртывание контейнера Amnezia, управление и
даже **подключение** full-access профилем идут поверх **SSH к серверу**, причём
разворачивание контейнерной сборки происходит **с IP пользователя**, который это
делает. У профиля полного доступа само подключение к VPN гейтится SSH-проверкой:
приложение сначала ходит по SSH и лишь при его доступности поднимает туннель. Нет
SSH-достижимости → **ошибка 305**, туннель не поднимается вовсе.

> **Дедлок, в котором мы оказались из-за прежней рекомендации.** Если SSH открыт
> ТОЛЬКО с адреса VPN-туннеля к этому же серверу, то: чтобы подключиться/управлять
> full-access профилем, нужен SSH; чтобы получить SSH, нужен поднятый туннель; а
> туннель full-access профиля не поднимется, пока SSH недоступен. Замкнутый круг —
> ни зайти, ни починить из приложения. Выход из него — SSH-доступ по **отдельному,
> независимому от этого туннеля** каналу (ваш белый IP / другой ваш VPN-выход).
> Именно поэтому вайтлист должен содержать ВАШИ адреса, а не адрес туннеля.

> Обычные (гостевые/клиентские) пиры этим НЕ страдают — поднимаются чисто по
> ключам, серверный SSH им не нужен. SSH-gate — свойство именно full-access профиля.

**Бот хранит ссылку полного доступа за вас (зашифрованно).** Чтобы не зависеть от
одного устройства, назначьте боту «устройство полного доступа»: при холодном
старте бот сам предложит выбрать его из устройств без профиля (пир обычно
называется `Admin [платформа]`) — или сделайте это позже через карточку такого
устройства → **«Прописать строку подключения»** → пришлите ссылку полного доступа.

- **Хранение только зашифрованное.** Ссылка = root к серверу; она шифруется тем
  же ключом, что и бэкапы (из env, пассфразу в Telegram бот не запрашивает).
  Поэтому назначить устройство полного доступа можно **только при включённом
  шифровании бэкапов** (`awg-bot reconfigure` → шифрование). Причина: БД уходит в
  бэкап через Telegram — plaintext-ссылка там раздала бы root через облако.
- **Метка `[Доступ к серверу]`**, привязка к клиенту «Администратор», всегда
  **одно** такое устройство (назначение второго предложит перенос с открепления
  первого).
- **Получить ссылку обратно** (новое устройство, переустановка приложения):
  карточка → **«Получить ссылку доступа»** → QR / ссылка / файл, каждый способ
  через диалог-предупреждение (root-доступ — не делитесь).
- **Замена** (пересоздали контейнер) и **снятие метки** — через явные
  предупреждения об утрате прежней ссылки (восстановить её из бота нельзя).

### B. Потерянный клиентский конфиг: восстановление и перевыпуск

Важно различать «извлечь ТОТ ЖЕ ключ» и «восстановить доступ»:

- **Тот же самый приватный ключ уже существующего пира заново получить нельзя** —
  в WireGuard/AWG сервер хранит только публичный ключ клиента; приватный жил лишь
  на самом устройстве в момент генерации. Если устройство, где он лежал, утрачено —
  этот ключ не восстановить ниоткуда.
- **Но с самого́ устройства, которое уже подключается**, свой конфиг вытащить
  можно: приватный ключ лежит в его локальном хранилище (иначе туннель бы не
  поднялся) — экспортируйте свою запись через шаринг в приложении.
- **Восстановить ДОСТУП** (не тот же ключ) — можно всегда через **перевыпуск**:
  в приложении Share → вкладка **Users** показывает клиентов с сервера, позволяет
  отозвать старого и выпустить нового; либо, что удобнее в нашей схеме, — **завести
  устройство заново через бота** (карточка клиента → добавить устройство): бот
  сгенерит новую пару и выдаст конфиг/QR/файл, доступные для повторного просмотра
  сколько угодно раз. Старый пир при этом отзывается/удаляется.

Практический вывод: раздавать и перевыпускать конфиги удобнее через бота — там они
всегда доступны повторно, и не нужно ловить «единственное окно» приложения.

### C. App-устройства (созданные в приложении, не ботом)

Такие пиры бот подхватывает реконсиляцией как **app-устройства**: видит pubkey и
считает трафик, но приватного ключа у него нет → выдать конфиг/QR по ним не может.
Если у вас есть `vpn://` именно этого устройства с его приватным ключом (например,
это профиль-1 с полным доступом), карточку app-устройства можно повысить до
полноценной: карточка → **«Прописать строку подключения»** → прислать ссылку. Бот
разбирает её тип автоматически: клиентская ссылка (с приватным ключом) — проверит,
что ключ деривит в pubkey пира (`WRONG_DEVICE` иначе), и заберёт устройство под
управление; ссылка полного доступа — сохранит устройство как «Доступ к серверу»
(см. раздел A). Для обычных клиентских app-устройств восстановление ключом
неприменимо (приватного ключа в экспорте нет) — их проще удалить и завести заново.
Осиротевшие пиры (пропавшие из живого конфига) бот убирает сам после порога сверок
(`missing_sweeps_threshold`).

## 12. Ручная установка (для справки)

```bash
mkdir -p /opt/awg-bot && cd /opt/awg-bot   # скопировать сюда проект
python3.12 -m venv venv                    # ЯВНО 3.12 (StrEnum)
./venv/bin/pip install -r requirements.txt
mkdir -p /etc/awg-bot/conf /var/lib/awg-bot
cp conf/*.yaml /etc/awg-bot/conf/          # вписать network.server_host и
                                           # network.server_port (app.yaml)
printf 'BOT_TOKEN=…\nADMIN_ID=…\n' > /etc/awg-bot/env && chmod 600 /etc/awg-bot/env
./venv/bin/python -m tools.manage_secrets  # шифрование резервных копий
./venv/bin/python -m awgbot                # проверить форграундом (или ./run.sh)
cp awg-bot.service /etc/systemd/system/ && systemctl daemon-reload
systemctl enable --now awg-bot
```

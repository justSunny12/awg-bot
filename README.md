# AmneziaWG Telegram-бот

Управление самостоятельно развёрнутым AmneziaWG VPN (докерная версия, контейнер
`amnezia-awg2`) через Telegram. Роли админ/клиент/друг, инвайты, периоды подписки
с блокировкой трафика при истечении, накопительная статистика, двусторонняя
синхронизация имён с приложением Amnezia, устройство полного доступа к серверу
(зашифрованное хранение root-ссылки), локальный мониторинг железа хоста.

Этот файл — обзор проекта и архитектуры. Подробности по установке, конфигурации
и эксплуатации — в **[docs/README-bot.md](docs/README-bot.md)**.

---

## Архитектура

Бот co-located с awg-контейнером на одном хосте и управляет им локально через
`docker exec`. Единственный внешний канал — Telegram Bot API.

```
┌──────────────────────────────────────────────────────────┐
│  Хост (AmneziaWG + бот на одной машине)                   │
│                                                          │
│  awg-bot (systemd)                                       │
│   • Telegram Bot API   ← единственный внешний канал       │
│   • SQLite (состояние)                                   │
│   • APScheduler        (опрос трафика, мониторинг)        │
│   • inotify-вотчдог     awg0.conf                         │
│        │ docker exec                                     │
│        ▼                                                 │
│  amnezia-awg2 (docker)                                    │
│   • awg0.conf / clientsTable                             │
│   • iptables-блокировки (внутри контейнера)              │
│                                                          │
│  метрики CPU/RAM/диск — бот читает сам: /proc + statvfs   │
└──────────────────────────────────────────────────────────┘
```

Почему только co-located: управление awg — это локальный `docker exec` (по сети
до контейнера бот не ходит), а сам бот выходит в Telegram через тот же хост.
Разносить бота на отдельную машину смысла нет — в РФ при заблокированном Telegram
такой бот терял бы и доступ к Telegram (нет туннеля), и к awg (чужой докер), то
есть не закрывал бы ни одной задачи. Метрики железа бот снимает локально, без
агентов и сетевых приёмников.

## Устройства и роли

**Устройство** = пир AmneziaWG. Тип определяется по факту (без ярлыков-прокладок):

- **управляемое** (`is_managed`) — у бота есть приватный ключ: может выдавать
  ссылку/QR/файл, передавать другу, менять лимит. Создано ботом или
  реставрировано клиентской `vpn://` ссылкой.
- **app** (`is_app`) — пир из приложения Amnezia, ключа у бота нет (подхвачен
  реконсиляцией). Помечается суффиксом `*`; можно «прописать строку» (реставрация)
  или удалить.
- **полного доступа** (`is_admin`) — несёт зашифрованную `vpn://` ссылку полного
  доступа к серверу (root). Метка `[Доступ к серверу]`, привязано к клиенту
  «Администратор», всегда одно. Бот лишь хранит ссылку и отдаёт её (QR/ссылка/файл)
  через диалог-предупреждение — приложение Amnezia разворачивает подключение само.

**Роли** (проставляет middleware по tg_id):

- **admin** (из конфига) — полный доступ ко всему.
- **client** — свой профиль: свои устройства (выдача конфигов, переименование,
  лимиты), передача устройства другу. Не может трогать чужое.
- **invited** (друг) — управляет только устройствами, которые ему расшарили
  (мультидружба: один tg_id → устройства от разных владельцев); переименование
  недоступно.

**Синхронизация имён — двусторонняя**, источник истины — последний, кто явно
переименовал: бот пишет в БД + clientsTable, приложение → подхватывается
реконсиляцией. Ловится для всех пиров.

**Устройство полного доступа**: при холодном старте, если оно не назначено, бот
подсвечивает это админу (выбрать из устройств без профиля или проигнорировать).
Ссылка хранится **только зашифрованной** (тем же ключом, что бэкапы — из env,
пассфраза в Telegram не запрашивается); без включённого шифрования назначить
нельзя. Замена/снятие метки/перенос — через явные предупреждения об утрате.

## Установка

Два файла рядом (`awg-bot.tgz` + `awg-bot-install.sh`), затем:

```bash
sudo bash awg-bot-install.sh          # интерактивная установка «в одно окно»
```

Установщик greenfield-only: разворачивает код в `/opt/awg-bot`, создаёт venv,
пишет конфиг/секреты, ставит systemd-сервис. Дальше управление — командой
`awg-bot` (`reconfigure|update|uninstall|backup|restore|status|logs`).

Топология подхватывается автоматически: внешний IP определяется сам
(`ip route get`), порт awg и VPN-подсеть — из живого контейнера (`detect_topology`
читает `awg0.conf`); пользователь лишь подтверждает Enter'ом. Опционально —
хардненинг SSH: `install/harden_firewall.sh` (точечный nftables, SSH-вайтлист
ваших IP + своя VPN-подсеть).

## Приостановка подписки («в отпуск»)

Только для годовой подписки. Пауза **замораживает время**: `period_end` сдвигается
вперёд на зарезервированные дни, VPN-доступ отключается (PAUSED-каскад на
клиента и устройства). Единственный лимит — **суммарный за период**
(`pause_max_total_days`, по умолчанию 28, `conf/pause.yaml`); ни «максимума за один
вход», ни ограничения остатком подписки нет. Списание по факту с округлением
вверх: 1 минута паузы = 1 день; неиспользованный остаток при досрочном выходе
возвращается в `period_end` и в лимит.

**Вход** (клиент, «Управлять подпиской»): выбор длительности кнопками
7/14/«весь остаток» (недоступные пресеты скрыты, дубли не показываются;
«✏️ Другое» — ввод числа 1..доступного, скрыто, когда вводить нечего) →
предупреждение о тупике (VPN выключится, выйти досрочно можно только через
бота; если Telegram доступен лишь через этот VPN — сам не выйдешь) →
подтверждение. Кнопка входа не прячется при нулевом остатке — по нажатию
объясняется: «Лимит дней приостановки в текущем периоде исчерпан». В карточке
подписки счётчик: `Приостановка: доступно N/M дн. до <конец подписки>`.

**Три выхода из паузы:**

1. **Кнопка «Возобновить»** в боте (клиент) — в любой момент.
2. **Админ**: в карточке клиента при активной паузе самой верхней кнопкой
   «▶️ Вывести из приостановки» — тот же пересчёт дней. Запасной выход, если
   клиент заперся.
3. **Аварийный e-mail-выход** (опционально): при входе в паузу клиенту выдаётся
   одноразовый код (8 симв., алфавит без похожих 0/O/1/l). Заперевшись без
   Telegram, клиент шлёт письмо с кодом **в теме** на настроенный адрес — бот
   опрашивает IMAP-ящик исходяще (портов на хосте не открываем), находит код,
   снимает паузу и отвечает «Код принят, доступ восстановлен». Свойства:
   `From` не является аутентификацией (спуфится) — секрет только код; тело
   письма не читается вообще; невалидный код игнорируется молча (письмо
   помечается прочитанным, ответа нет); код одноразовый и живёт только пока
   активна пауза. Настройка — целиком через визард
   (`awg-bot reconfigure` → «Настроить аварийный email-выход»): адрес ящика
   (для `@icloud.com` IMAP/SMTP подставляются сами, иначе ручной ввод),
   app-specific пароль, опциональный алиас (Enter — сам адрес ящика), интервал
   опроса (не менее 60 сек). Секреты — в env, остальное — `conf/email.yaml`; пустые
   поля = фича спит.

Авто-выход по истечении зарезервированного срока делает watchdog
(`check_pauses`); бессрочную админскую паузу (admin_open) снимает только админ.

## Модель конфигурации и безопасности

- **Топология и правила — в yaml** (`/etc/awg-bot/conf`): источник истины, генерит
  установщик, правится руками на холодную, переживает обновление кода.
- **Секреты — только в env** (`/etc/awg-bot/env`, `600`). Недостающие обязательные
  ключи ловит `config.validate()` при старте с адресным сообщением.
- **Крипто-материал AWG** (ключ сервера, psk, обфускация awg-протокола, порт)
  читается **live из контейнера** и кэшируется — второго источника истины нет.
- **Приватные ключи управляемых устройств** живут в БД (`/var/lib/awg-bot`);
  единственная копия вне сервера — **шифрованный бэкап** (SecretBox), если задан
  ключ/пассфраза.
- **Ссылка полного доступа** (root к серверу) хранится в БД **только
  зашифрованной** тем же ключом, что и бэкап (из env). Причина: БД уходит в бэкап
  через Telegram — plaintext-ссылка там раздала бы root через облако. Без
  включённого шифрования назначить устройство полного доступа нельзя.
- **Клиент «Администратор»** (устройства админа + полного доступа) неудаляем и не
  ограничивается (блок/лимит/пауза недоступны, defense-in-depth в сервисе).
- **Хардненинг**: `install/harden_firewall.sh` (точечный nftables — только
  SSH-вайтлист, общую политику не трогаем), секреты в env `600`.
- **Время везде UTC+3.** Сервер живёт в UTC, бот конвертирует.

## План интеграционного теста на сервере (перед боевым запуском)

Юнит-логика и парсеры проверены против реальных данных. На сервере нужно
проверить сквозные цепочки (то, что нельзя проверить без контейнера):

1. `./run.sh` вручную (форграунд, = `python -m awgbot`) — стартует без ошибок,
   вотчдог подключается (лог «Вотчдог подключён к /proc/…»). Если вотчдога нет —
   inotify через `/proc/<PID>/root` не сработал (namespace); подхват изменений
   идёт страховочной mtime-сеткой (~10 сек).
2. `/start` от админа → меню. Создать тестового клиента → инвайт-ссылка.
3. Активировать инвайт со второго аккаунта → добавить устройство → подключиться
   телефоном (проверка генерации в бою).
4. В приложении Amnezia добавить устройство → бот шлёт админу «обнаружено новое
   устройство» (вотчдог + реконсиляция).
5. Продлить/сократить срок клиента до «пары минут», дождаться истечения →
   интернет пропадает, туннель жив; продлить → вернулся.
6. `docker restart amnezia-awg2` → бот на следующем цикле мониторинга переналагает
   блокировки и ребайндит вотчдог.
7. Проверить метрики хоста в панели (кнопка «Статус» форсит live-снимок) и
   алерт загрузки (порог/стрик из `resource_alerts.yaml`).
8. Устройство полного доступа: включить шифрование бэкапов (`reconfigure`), при
   старте выбрать пир `Admin [...]` из устройств без профиля, прислать ссылку
   полного доступа → получить её обратно (QR/файл) через диалог-предупреждение;
   проверить снятие метки и перенос.

## Структура (пакет по слоям)

Направление зависимостей: `core ← util ← infra ← domain ← bot ← runtime`,
`config` доступен всем. Запуск — `python -m awgbot` (обёртка `__main__.py`);
форграунд — `./run.sh`.

```
awgbot/
  core/      config     чтение yaml + секреты из env, validate()
             enums, models, blocks   доменные типы, битмаски блокировок
  util/      timeutil   время UTC+3, форматтеры, формат даты Amnezia
             qrgen      QR/GIF; secrets_util  argon2id/SecretBox (крипто бэкапов)
  infra/     db         SQLite (per-thread), схема, доступ
             awg        ЕДИНСТВЕННЫЙ слой docker exec (парсеры + операции + detect_topology)
             email_resume  IMAP/SMTP-механика аварийного email-выхода из паузы + генерация кода
  domain/    services   бизнес-логика (склейка db+awg+configgen)
             configgen  генерация vpn:// и .conf; classify_vpn_link (client/full_access)
  bot/       texts, keyboards, guides   презентация
             filters, middleware, callbacks, states   glue aiogram
             notifier   рассылка уведомлений (пейсинг + RetryAfter)
             handlers/  роутеры client + admin + friend + …
  runtime/   main         сборка и запуск
             scheduler    APScheduler-задачи (в т.ч. поллер email-выхода)
             watcher      inotify-вотчдог (/proc/<PID>/root, дебаунс)
             hostmetrics  локальные метрики железа (/proc + statvfs)
  assets/    guides/      скриншоты пошаговых гайдов (смена региона App Store)

tools/       manage_secrets  шифрование бэкапов; restore_backup  расшифровка (*.enc)
install/     awg-bot-install.sh (внешний bootstrap), harden_firewall.sh
conf/        *.yaml — конфигурация; run.sh — форграунд-запуск
```

## Тесты

Пакет покрыт тестами четырёх уровней (`tests/`, pytest):

```
tests/unit/         чистая логика без БД/сети (timeutil, configgen, secrets_util, hostmetrics, awg-парсеры)
tests/smoke/        импорт всего пакета, config.validate(), сборка aiogram-роутеров
tests/integration/  настоящая временная SQLite + фейковый awg-слой (db, services)
tests/e2e/          сквозные сценарии: жизненный цикл, лимиты трафика, пауза/грейс, мониторинг
```

```bash
./venv/bin/pip install -r requirements-dev.txt
./venv/bin/python -m pytest              # весь набор
./venv/bin/python -m pytest -m unit      # один уровень (unit|smoke|integration|e2e)
```

Тесты не трогают docker/Telegram: awg-слой подменяется in-memory фейком, БД —
временный файл. Боевой таргет — Python 3.12.
